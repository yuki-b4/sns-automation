## 2. テーマ別評価

### ① [子の記憶に残らない父]（2026-05-07）

### ② [結婚意識で冷める瞬間の正体]（2026-05-09）

- **改善点**: 「正体」「脳内反応」「神経設計図」など類似ラベルが7本以上提案されており、今後は同系統内での切り口分化（時間損失視点 / 神経反応視点 / パートナー視点）が必要。target_brain 未記載のため、次回採択時はラベリングを徹底したい。

### ③ [優しさが枯渇する瞬間の構造]（2026-05-10、target_brain=両方）

## 3. 採択シグナル分析（運用者キュレーション）

未採択提案 79本の傾向から、運用者が**選ばない**切り口の共通点：

1. **同一テーマの過剰反復は通らない**: 「結婚意識で冷める」系・「自責と他責の往復」系・「心理学知識が応用できない」系・「別れ際の言葉」系は各4〜7本提案されているが、各系統につき採択は1本以内。提案側の自己重複が採択率を下げている。
2. **爬虫類脳・単独訴求の未採択率が顕著**: 未採択 79本のうち想定刺激=爬虫類脳の提案が約30本と最多。「神経設計図」「内側回路」「脳内反応」等の身体・神経メタファー単独は採択されにくい傾向。
3. **抽象度の高すぎる構造語ラベル**は通りにくい印象がある（「自己破壊」「窒息感」「複利損失」など強い比喩単発のもの）。一方、採択されたのは「子の記憶に残らない父」「結婚意識で冷める瞬間」のように**具体的なシーン＋構造**の組み合わせ。

### generate_note.py への含意
- 同一 theme_label 系統の連続提案を抑制し、3提案間で**シーン軸 / 時間軸 / 関係性軸**のいずれかを必ず変える制約をプロンプトに足すと採択率が上がる可能性。
- 爬虫類脳単独提案は「身体反応」よりも「時間・選択肢の不可逆損失」に寄せたほうが採択されやすい仮説。

## 4. Threads → note の一貫性評価

### テーマ一貫性
直近 Threads 投稿（dialogue / structure / personal / opinion / permission）と、投稿済み note 4本のテーマは**高い一貫性**がある：

- Threads「『いい人なんだけど』って言われたあの日から」「関係を壊す直前にやってたこと」 ⇔ note「結婚意識で冷める瞬間の正体」「踏み込みと見極めの停止状態」
- Threads「相手を大切にしようとして苦しくなったとき」（permission） ⇔ note「優しさが枯渇する瞬間の構造」
- Threads「エンジニアからコーチに転職した初日」（personal） ⇔ note「子の記憶アルバム」（父親としての時間軸）

Threads で**問いを開いた話題が note でテーマ化される**流れは概ね機能している。

### handoff ルールの実践しやすさ
`engagement_design_rules.threads_to_note_handoff`（Threads 投稿の問いを note 冒頭 300 字以内で明示的に回収）について、現状は手書きフェーズで実践可能な粒度ではあるが、**どの Threads 投稿を回収したかをレコード化する仕組みが note 投稿DB 側にない**ため、運用者の記憶依存になっている。後述（§6）。

## 5. 来週の推奨アクション

### 重点的に伸ばすべき切り口
1. **「優しさの構造化」系**: 採択された [優しさが枯渇する瞬間の構造] は「両方」訴求の数少ない成功例。優しさ × 構造的説明はエンジニア／PM ペルソナの読解スタイルと噛み合う。
2. **「子・家族の不可逆時間」系**: [子の記憶に残らない父] のように**時間損失を具体的シーンで描く**切り口は、爬虫類脳（損失）と哺乳類脳（所属）の同時起動が容易。
3. **「結婚意識で冷める」系の派生**: 採択実績あり。ただし同系統提案が過剰なので、次回は「冷めた側」ではなく「冷められた側の視点」など視点反転で1本に絞る。

### 避けるべき／重複気味の切り口
- **「自責と他責の往復」系**（提案4本／採択0）
- **「心理学知識が応用できない」系**（提案4本／採択0）
- **「別れ際の言葉が残響する」系**（提案5本／採択0）
- 抽象比喩単発（「窒息」「複利損失」「在庫切れ」等）の単独訴求

これらは Threads で問いを試してから note 化するか、もしくは提案プールから一旦外して再供給を止める。

### target_brain 分散の調整
投稿済み4本中 target_brain 明記は1本（両方）のみ。**まず採択時に target_brain をレコードに必ず記入する運用**を優先。その上で、来週は「爬虫類脳単独」「哺乳類脳単独」を各1本ずつ採択して、両方ばかり通す傾向の検証を行いたい。

### テーマ × target_brain の試行案
1. **[既婚同期との会話が噛み合わない夜] × 哺乳類脳**: 所属欲求の歪みを扱う Threads dialogue 系と接続しやすい。
2. **[婚期逃しの選択肢複利損失] × 爬虫類脳**: 時間損失を数値・選択肢の不可逆性で描く。「複利」はエンジニア親和性が高い。
3. **[条件リストが機能しない理由] × 両方**: Threads opinion「理想の相手リスト＝要件定義書」と直結。エンジニアメタファーで橋渡しできる。

## 6. 手書きnote本文ルールへの追加提案

- **採択時に「回収する Threads 投稿の type タグ」を 1 つメモする運用**: 現状の note 投稿DB には対応 Threads を紐づける列がない。本文を書く前に "dialogue を回収する" "structure を回収する" のどちらの handoff かを宣言してから書き始めると、冒頭 300 字の回収設計が安定する。
- **「構造」型タイトルが採択されたら、本文も先に構造図解を1段落入れる**: [優しさが枯渇する瞬間の構造] のように「構造」を含むテーマが通った場合、Before-After よりも先に**現象の分解（A→B→C の連鎖）**を提示する流れが、エンジニア／PM ペルソナの読解と整合しやすい。
- **採択した theme_label と本文ラスト1段落のキーワードを揃える**: DB の title は改題されるが theme_label は提案時の核を保持している。読了後アクション設計の手前で theme_label のキーワードを1回必ず置くと、再訪時の検索性が上がる。
- **同一テーマ系統で2本目を書くときは「視点反転」を 1 行目で宣言**: 結婚意識で冷める系のように同系統内で複数本書く場合、冒頭で「前回は冷めた側を書いた。今回は冷められた側から見る」など立ち位置の差分を 1 行で示すと、読者の既読感を回避できる。
